# Progressive Sensemaking

Progressive sensemaking is the runtime discipline for using a SkillSpec without
feeding the whole skill file into the model context.

The goal is not to hide the spec from the agent. The goal is to make the agent
read the right part at the right time:

1. map the shape;
2. decide the active route;
3. query the active ids;
4. follow references;
5. load only required imports and resources;
6. align the trace after work is complete.

This is how SkillSpec turns a large skill from a prose blob into a navigable
contract.

## Why Sensemaking Exists

A large skill can contain routes, rules, dependencies, imports, resources, code,
recipes, states, tests, proof expectations, and trace requirements. Loading all
of that at once creates three problems:

- the model may spend context on inactive material;
- the agent may treat examples or resources as active instructions;
- reviewers lose the connection between the task, selected route, matched rules,
  and executed work.

`skillspec sensemake`, `skillspec decide`, `skillspec query`, and
`skillspec refs` address those problems by giving the agent a navigation layer.

The loader generated by `skillspec compile` tells agents to use this navigation
layer. The implementation says the same thing in the sensemake escalation list:

- start with `sensemake --view index` only when unfamiliar;
- use `decide` for task routing;
- use `query` and `refs` for matched ids instead of reading the whole YAML;
- escalate from index to summary to full only when needed.

## The Sensemaking Commands

### `sensemake`

`skillspec sensemake <spec> --view index` produces a map of the spec:

- spec id, title, path, and view;
- section roles;
- section counts;
- ids for the high-signal sections;
- navigation hints;
- escalation guidance.

The index view includes:

- `routes`: strategy choices;
- `rules`: steering logic;
- `states`: lifecycle phases;
- `dependencies`: static substrate;
- `commands`: executable templates;
- `recipes`: ordered procedures;
- `closures`: completion obligations;
- `tests`: behavior checks.

The summary and full views add:

- `trace`: decision evidence;
- `proof`: verification metrics;
- `imports`: lazy external context;
- `resources`: local reference material.

That shape is intentional. Imports and resources can be large, so the index view
does not start there.

### `decide`

`skillspec decide <spec> --input '<task>'` evaluates the routing contract. When
called with `--trace-dir`, it also records decision evidence.

The decision result is the runtime entry point. It can include:

- selected `route`;
- route selection basis;
- `route_order`;
- selected route `execution_plan`;
- `forbid`;
- `allow`;
- `elicit`;
- `after_success`;
- `matched_rules`;
- decision `reason`.

An agent should not act from route labels alone. It should read the decision JSON
and then query the active route, matched rules, dependencies, commands, recipes,
imports, states, or closures that the decision references.

### `query`

`skillspec query <spec> <handle>` retrieves a precise part of the spec.

Supported collection handles include:

- `routes`;
- `rules`;
- `states`;
- `dependencies`;
- `commands`;
- `recipes`;
- `closures`;
- `tests`;
- `trace`;
- `proof`;
- `imports`;
- `resources`;
- `code`;
- `artifacts`;
- `snippets`;
- `elicitations`.

Supported item handles include:

- `route:<id>`;
- `rule:<id>`;
- `state:<id>`;
- `dependency:<id>`;
- `command:<id>`;
- `recipe:<id>`;
- `closure:<id>`;
- `test:<name>`;
- `import:<id>`;
- `resource:<id>`;
- `code:<id>`;
- `artifact:<id>`;
- `snippet:<id>`;
- `elicitation:<id>`.

Dot-field projection is supported. Examples:

```sh
skillspec query skill.spec.yml rule:cli_invocations_use_rote_exec.forbid
skillspec query skill.spec.yml command:deps_check.requires
skillspec query skill.spec.yml route:browser_handoff.handoff
skillspec query skill.spec.yml test:browse_selects_browser.expect
```

Quote the handle when a test name contains spaces.

When a field path is requested, the implementation projects from the full value
for that target. That lets an agent ask for exactly the field it needs without
loading the whole file.

### `refs`

`skillspec refs <spec> <handle>` retrieves outgoing relationships for item
handles.

The current implementation supports outgoing refs for:

- `rule:<id>`;
- `command:<id>`;
- `state:<id>`;
- `recipe:<id>`;
- `route:<id>`;
- `test:<name>`.

Examples:

```sh
skillspec refs skill.spec.yml rule:external_service_tasks_are_adapter_first
skillspec refs skill.spec.yml command:deps_check
skillspec refs skill.spec.yml route:shell_then_browser_handoff
skillspec refs skill.spec.yml test:browse_selects_browser
```

Refs are useful because they turn a chosen id into the next ids to inspect:

- a rule can point to preferred routes, route order, forbids, elicitations, and
  after-success actions;
- a command can point to dependencies, files, environment variables, and auth
  requirements;
- a state can point to `next`, `yes`, and `no` states;
- a recipe can point to imports, resources, dependencies, artifacts, and step
  targets;
- a route can point to checks, handoff targets, execution-plan owner skills,
  phase routes, phase handoffs, and jump targets.

## Recommended Agent Workflow

Use this workflow for a SkillSpec-backed task.

### 1. Start With The Loader

Load `SKILL.md` only to discover the spec and the runtime contract. Do not treat
loader prose as the behavioral source of truth.

If the loader says a decision is required before tools, obey that gate. Until the
decision is read, task actions should be limited to loading the spec and running
SkillSpec navigation or decision commands for that spec.

### 2. Map The Spec Shape

If the spec is unfamiliar, run:

```sh
skillspec sensemake ./skill.spec.yml --view index
```

Use the result to learn which sections exist and which ids are available. Do not
start by reading imports, resources, or the full YAML.

### 3. Decide The Active Contract

Run:

```sh
skillspec decide ./skill.spec.yml --input '<user task>' --trace-dir .skillspec/traces
```

Strip skill invocation prefixes before passing the task input when the loader
says to do so.

Read the decision JSON. Preserve the `run_dir` when a trace is emitted.

### 4. Query The Selected Route And Matched Rules

Query the selected route:

```sh
skillspec query ./skill.spec.yml route:<route-id> --view summary
skillspec refs ./skill.spec.yml route:<route-id> --view summary
```

For each matched rule:

```sh
skillspec query ./skill.spec.yml rule:<rule-id> --view summary
skillspec refs ./skill.spec.yml rule:<rule-id> --view summary
```

Build the active checklist from the decision and these query results:

- selected route;
- execution plan and phase order;
- matched rule reasons;
- forbids;
- allows;
- elicitations;
- after-success obligations;
- handoff boundaries;
- jump points.

### 5. Check Dependencies Before Execution

If a command or route leads to a command id, inspect the command requirements:

```sh
skillspec query ./skill.spec.yml command:<id>.requires
skillspec refs ./skill.spec.yml command:<id>
skillspec deps check ./skill.spec.yml --command <id>
```

For route-level or recipe-level work, inspect the route or recipe refs first,
then check the relevant dependencies.

The agent should not substitute another tool just because it is available in the
harness. If the active contract forbids a substitution, the agent must stop,
ask, or choose an allowed route.

### 6. Load Imports Only When Required

Imports are runtime-loadable instruction material. Load an import when:

- it is marked `load: always`;
- the active route, rule, recipe, command, code path, or refs require it;
- the loader or harness has a declared reason to inspect it.

Resources are different. A resource is provenance or supporting material unless
the spec connects it to active behavior.

### 7. Execute Through The Harness

Once the active contract is materialized, the harness decides which tool actions
are allowed. SkillSpec can declare commands, dependencies, safety classes,
handoffs, and proof expectations. It does not bypass the harness approval model.

Execution should preserve structured evidence whenever possible, especially for
commands, browser work, network work, and file changes.

### 8. Align After Completion

After a traced decision, run:

```sh
skillspec trace align ./skill.spec.yml \
  --decision-trace <run_dir> \
  --summary \
  --proof-digest <run_dir>/proof-digest.json
```

If structured execution evidence exists, include it:

```sh
skillspec trace align ./skill.spec.yml \
  --decision-trace <run_dir> \
  --execution-trace <jsonl> \
  --summary \
  --proof-digest <run_dir>/proof-digest.json
```

Decision replay can prove deterministic steering behavior. Execution obligations
may still be unproven if the harness did not provide execution evidence.
When several proof rows remain, use the digest to build one final-proof JSONL
file and append it with
`skillspec progress batch <run_dir> --file <run_dir>/final-proof.jsonl
--checkpoint "checkpointing evidence" --summary` before one final alignment
rerun.

## Escalation Rules

Escalate only when needed:

- `index`: use for orientation and ids.
- `summary`: use for active route, rule, command, recipe, state, or dependency
  details.
- `full`: use for authoring, debugging, schema issues, complex conflicts, or
  when a field projection is not enough.
- direct file read: use only when the CLI is unavailable or the task is to edit
  or debug the spec itself.

This escalation model is not a secrecy boundary. It is a context and reliability
boundary. The agent can read the full file when necessary, but should not make
that the default runtime behavior.

## Common Failure Modes

Avoid these mistakes:

- reading the whole YAML before deciding the active route;
- reading all imports because they exist;
- treating resources as active instructions;
- acting from a route label without reading the decision JSON;
- ignoring matched-rule forbids;
- skipping `refs` and missing dependencies, elicitations, handoffs, or jumps;
- using an undeclared harness tool as a convenient substitute;
- reporting success without after-success obligations or alignment status.
